114 ////////////////////////////////////////////////////////////////////////////////
115 // ValueType implementation.
116 //
117 // We represent ValueTypes as bit fields. We store a mix of spec-level
118 // information and additional caches that are useful for efficient algorithms
119 // (mostly of subtyping). The key design consideration is that we need to
120 // express many different variations of attributes, checks, switches; so we
121 // want a ValueType representation that efficiently supports many different
122 // kinds of queries and classifications.
123 // The usage of the bit field is as follows:
124 // bits 0-1: TypeKind: numeric, indexed ref, generic ref, or sentinel. The bits
125 // are chosen such that they can also be interpreted as:
126 // bit 0: is it a reference of some sort?
127 // bit 1: if it's a reference: does it have an index?
128 // if it isn't a reference: is it an internal sentinel?
129 //
130 // bit 2: if it's a reference, is it nullable? Otherwise: unused.
131 // bit 3: if it's a reference, is it an exact reference? Otherwise: unused.
132 //
133 // For indexed (aka user defined) types, the next two fields cache information
134 // that could also be looked up in the module's definition of the type:
135 // bit 4: is it a shared type? Note that we consider numeric types as shared,
136 // i.e. "shared" means "safe to use in a shared function".
137 // bits 5-7: RefTypeKind: if it's a reference (indexed or generic), what
138 // category of type (struct/array/function/continuation) is it?
139 // Non-reference types store the value "kOther".
140 //
141 // bits 8-27: For indexed types, the index. For other types, the StandardType.
142 // The "ValueType" subclass stores module-specific indices, the
143 // "CanonicalValueType" stores canonicalized indices.
144 // For non-indexed types, we also define their respective
145 // "NumericKind" or "GenericKind", which in addition to the right
146 // StandardType include the other relevant lower bits.
147 //
148 // With these assignments, it turns out that "HeapType" is effectively a
149 // subset of "ValueType", where bits 0-1 are known to *not* be "numeric",
150 // and bit 2 (which is ValueType-specific) is ignored.

267 enum class StandardType : uint8_t {

273 // We additionally define separate enums for generic and numeric types, so that
274 // we can write exhaustive switch statements. These enums' values include
275 // additional relevant bits in the {bit_field_}s of the respective HeapTypes/

282  FOREACH_GENERIC_TYPE(DEF_ENUM)
283 #undef DEF_ENUM

304 // Useful for dense lookup tables.
305 constexpr uint32_t ToZeroBasedIndex(NumericKind kind) {
306  // Inside {StandardType}, generic types come first.
307  static_assert(static_cast<uint32_t>(StandardType::kI32) >=
308  kNumberOfGenericKinds);
309  uint32_t raw = PayloadField::decode(static_cast<uint32_t>(kind));
310  DCHECK_GE(raw, kNumberOfGenericKinds);
311  // As an additional safety net, as long as we happen to have exactly 8
312  // numeric types, we can conveniently apply a mask here. If we need to
313  // accommodate a different number of numeric kinds in the future, we should
314  // consider adding bounds checks at use sites.
315  static_assert(kNumberOfNumericKinds == 8);
316  return (raw - kNumberOfGenericKinds) & 0x7;
317 }
