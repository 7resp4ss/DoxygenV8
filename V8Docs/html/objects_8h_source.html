919 // This is an RAII helper class to emit a store-store memory barrier when
920 // publishing objects allocated in the shared heap.
921 //
922 // This helper must be used in every Factory method that allocates a shared
923 // JSObject visible user JS code. This is also used in Object::ShareSlow when
924 // publishing newly shared JS primitives.
925 //
926 // While there is no default ordering guarantee for shared JS objects
927 // (e.g. without the use of Atomics methods or postMessage, data races on
928 // fields are observable), the internal VM state of a JS object must be safe
929 // for publishing so that other threads do not crash.
930 //

932 // objects. It only ensures the weaker "do not crash the VM" guarantee.
933 //

935 // concurrent marking is active, field accesses are performed with relaxed
936 // atomics, and TSAN is unable to detect data races in shared JS objects. When
937 // concurrent marking is inactive, unordered publishes of shared JS objects in
938 // JS code are reported as data race warnings by TSAN.

940  public:

942  // A release fence is used to prevent store-store reorderings of stores to
943  // VM-internal state of shared objects past any subsequent stores (i.e. the
944  // publish).
945  //
946  // On the loading side, we rely on neither the compiler nor the CPU
947  // reordering loads that are dependent on observing the address of the
948  // published shared object, like fields of the shared object.
949  std::atomic_thread_fence(std::memory_order_release);
