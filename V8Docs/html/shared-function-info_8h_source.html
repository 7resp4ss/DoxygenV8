367  // For the sandbox, the function's data is split across two fields, with the
368  // "trusted" part containing a trusted pointer and the regular/untrusted part
369  // containing a tagged pointer. In that case, code accessing the data field
370  // will first load the trusted data field. If that is empty (i.e.
371  // kNullIndirectPointerHandle), it will then load the regular field. With
372  // that, the only racy transition would be a tagged -> trusted transition
373  // (one thread may first read the empty trusted pointer, then another thread
374  // transitions to the trusted field, clearing the tagged field, and then the
375  // first thread continues to load the tagged field). As such, this transition
376  // is only allowed on the main thread. From a GC perspective, both fields
377  // always contain a valid value and so can be processed unconditionally.
378  // Only one of these two fields should be in use at any time and the other
379  // field should be cleared. As such, when setting these fields use
380  // SetTrustedData() and SetUntrustedData() which automatically clear the
381  // inactive field.
382  // TODO(chromium:1490564): try to merge these two fields back together, for
383  // example by moving all data objects into trusted space.

385  WriteBarrierMode mode = UPDATE_WRITE_BARRIER);
