25 /**
26  * The V8 Sandbox.
27  *
28  * When enabled, V8 reserves a large region of virtual address space - the
29  * sandbox - and places most of its objects inside of it. It is then assumed
30  * that an attacker can, by exploiting a vulnerability in V8, corrupt memory
31  * inside the sandbox arbitrarily and from different threads. The sandbox
32  * attempts to stop an attacker from corrupting other memory in the process.
33  *
34  * The sandbox relies on a number of different mechanisms to achieve its goal.
35  * For example, objects inside the sandbox can reference each other through
36  * offsets from the start of the sandbox ("sandboxed pointers") instead of raw
37  * pointers, and external objects can be referenced through indices into a
38  * per-Isolate table of external pointers ("sandboxed external pointers").
39  *
40  * The pointer compression region, which contains most V8 objects, and inside
41  * of which compressed (32-bit) pointers are used, is located at the start of
42  * the sandbox. The remainder of the sandbox is mostly used for memory
43  * buffers, in particular ArrayBuffer backing stores and WASM memory cages.
44  *
45  * As the embedder is responsible for providing ArrayBuffer allocators, V8
46  * exposes the virtual address space backing the sandbox to the embedder.
47  */

55  // | (front) | Region : any other sandboxed objects. | (back) |

57  // ^ ^
58  // base end

100  /**
101  * Returns true if this sandbox is a partially-reserved sandbox.
102  *
103  * A partially-reserved sandbox is backed by a virtual address space
104  * reservation that is smaller than its size. It also does not have guard
105  * regions surrounding it. A partially-reserved sandbox is usually created if
106  * not enough virtual address space could be reserved for the sandbox during
107  * initialization. In such a configuration, unrelated memory mappings may end
108  * up inside the sandbox, which affects its security properties.
109  */
110  bool is_partially_reserved() const { return reservation_size_ < size_; }
111 
112  /**
113  * Returns true if the first four GB of the address space are inaccessible.
114  *
115  * During initialization, the sandbox will also attempt to create an
116  * inaccessible mapping in the first four GB of the address space. This is
117  * useful to mitigate Smi<->HeapObject confusion issues, in which a (32-bit)
118  * Smi is treated as a pointer and dereferenced.

170  /**
171  * Returns true if the given address lies within the sandbox address space.
172  */
173  bool Contains(Address addr) const {
174  return base::IsInHalfOpenRange(addr, base_, base_ + size_);
175  }

184  /**
185  * Returns true if the given address lies within the sandbox reservation.
186  *
187  * This is a variant of Contains that checks whether the address lies within
188  * the virtual address space reserved for the sandbox. In the case of a
189  * fully-reserved sandbox (the default) this is essentially the same as
190  * Contains but also includes the guard region. In the case of a
191  * partially-reserved sandbox, this will only test against the address region
192  * that was actually reserved.
193  * This can be useful when checking that objects are *not* located within the
194  * sandbox, as in the case of a partially-reserved sandbox, they may still
195  * end up in the unreserved part.
196  */
197  bool ReservationContains(Address addr) const {
198  return base::IsInHalfOpenRange(addr, reservation_base_,
199  reservation_base_ + reservation_size_);
200  }
