60 // CPU Registers.
61 //
62 // 1) We would prefer to use an enum, but enum values are assignment-
63 // compatible with int, which has caused code-generation bugs.
64 //
65 // 2) We would prefer to use a class instead of a struct but we don't like
66 // the register initialization to depend on the particular initialization
67 // order (which appears to be different on OS X, Linux, and Windows for the
68 // installed versions of C++ we tried). Using a struct permits C-style
69 // "initialization". Also, the Register objects cannot be const as this
70 // forces initialization stubs in MSVC, making us dependent on initialization
71 // order.
72 //
73 // 3) By not using an enum, we are possibly preventing the compiler from
74 // doing certain constant folds, which may significantly reduce the
75 // code generated for some assembly instructions (because they boil down
76 // to a few constants). If this is a problem, we could change the code
77 // such that we use an enum in optimized mode, and the struct in debug
78 // mode. This way we get the compile-time error checking in debug mode
79 // and best performance in optimized code.

185 // A few double registers are reserved: one as a scratch register and one to
186 // hold 0.0.
187 // f28: 0.0
188 // f30: scratch register.
189 
190 // V8 now supports the O32 ABI, and the FPU Registers are organized as 32
191 // 32-bit registers, f0 through f31. When used as 'double' they are used
192 // in pairs, starting with the even numbered register. So a double operation
193 // on f0 really uses f0 and f1.
194 // (Modern mips hardware also supports 32 64-bit registers, via setting
195 // (privileged) Status Register FR bit to 1. This is used by the N32 ABI,
196 // but it is not in common use. Someday we will want to support this in v8.)
197
