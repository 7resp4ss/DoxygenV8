54 // Builtins are additionally split into tiers, where the tier determines the
55 // distance of the builtins table from the root register within IsolateData.

60 // - T1 builtins have no distance guarantees.

299  /* Trampolines called when returning from a deoptimization that expects */ \
300  /* to continue in a JavaScript builtin to finish the functionality of a */ \
301  /* an TF-inlined version of builtin that has side-effects. */ \
302  /* */ \
303  /* The trampolines work as follows: */ \
304  /* 1. Trampoline restores input register values that */ \
305  /* the builtin expects from a BuiltinContinuationFrame. */ \
306  /* 2. Trampoline tears down BuiltinContinuationFrame. */ \
307  /* 3. Trampoline jumps to the builtin's address. */ \
308  /* 4. Builtin executes as if invoked by the frame above it. */ \
309  /* 5. When the builtin returns, execution resumes normally in the */ \
310  /* calling frame, processing any return result from the JavaScript */ \
311  /* builtin as if it had called the builtin directly. */ \
312  /* */ \
313  /* There are two variants of the stub that differ in their handling of a */ \
314  /* value returned by the next frame deeper on the stack. For LAZY deopts, */ \
315  /* the return value (e.g. rax on x64) is explicitly passed as an extra */ \
316  /* stack parameter to the JavaScript builtin by the "WithResult" */ \
317  /* trampoline variant. The plain variant is used in EAGER deopt contexts */ \
318  /* and has no such special handling. */ \
